# Phase 0 Research: 시민 참여형 스마트시티 안전 플랫폼

**Feature**: 001-civic-safety-platform | **Date**: 2026-06-19

명세(spec.md)와 클러리피케이션 결정을 바탕으로 기술 선택의 미해결 지점을 해소한다. 각 항목은 결정/근거/대안 형식으로 기록한다.

---

## R1. 멀티테넌트 데이터 격리 방식

- **Decision**: PostgreSQL **Row-Level Security(RLS) + 모든 테넌트 종속 테이블에 `tenant_id` 컬럼**을 사용한 단일 DB 논리 격리. 인증 컨텍스트(담당자 JWT의 tenant claim)로 RLS 정책을 강제한다.
- **Rationale**: 파일럿(단일 지자체)→다수 확장을 코드 변경 없이 수용(FR-026/027). 스키마-퍼-테넌트나 DB-퍼-테넌트보다 운영·마이그레이션 비용이 낮고, Supabase RLS와 정합. `supabase-postgres-best-practices` 스킬을 참조해 정책을 설계한다.
- **Alternatives**: 스키마 분리(테넌트 수 증가 시 마이그레이션 폭증), 물리 DB 분리(소수 대형 테넌트엔 유효하나 초기 비용 과다) → 기각.

## R2. 위험 분류 AI(VLM) 신뢰도 처리

- **Decision**: 경량 **객체탐지(YOLO 계열)로 후보 영역·객체** 추출 → **VLM(시각-언어)으로 위험 유형 분류 + 신뢰도 점수** 산출. 신뢰도 임계값 미만이면 후보 유형을 시민에게 제시하고 선택/수정(FR-004). 임계값 이상이면 자동 분류 후 시민 확인 단계 생략 가능("3초 신고").
- **Rationale**: SC-003(정확도 85%)·SC-002(3초) 동시 충족. 저화질/야간 등 엣지에서 강제 분류를 피해 오인식 행정 낭비를 줄임(부정적 영향 완화).
- **Alternatives**: 단일 분류 CNN(설명 가능성·다중유형 약함), 순수 LLM-only(지연·비용 과다) → 기각.

## R3. 개인정보 비식별(PII) 파이프라인

- **Decision**: 업로드 즉시 **얼굴·차량 번호판 탐지 → 블러 처리한 사본만 저장**, 원본은 저장하지 않는다(메모리 처리 후 폐기). `image-edit` 스킬 패턴을 참조하여 블러 단계를 구현.
- **Rationale**: SC-009(비식별 100%)·FR-008 충족, 초상권/PII 리스크(명세 핵심 장벽) 원천 차단.
- **Alternatives**: 원본+마스킹 메타데이터 보관(유출 위험), 사후 일괄 비식별(노출 창 존재) → 기각.

## R4. 지리 라우팅(관할 지자체 판별)

- **Decision**: **PostGIS**에 지자체 관할 경계(폴리곤)를 저장하고, 신고 좌표를 `ST_Contains`로 매칭해 테넌트로 라우팅(FR-028). 경계 외는 관할 외 안내.
- **Rationale**: 좌표→행정구역 매핑을 표준 공간연산으로 정확·고속 처리. 지도 표시(MapLibre)와 동일 좌표계 사용.
- **Alternatives**: 외부 지오코딩 API 의존(지연·비용·오프라인 불가), 행정코드 수기 매핑(경계 정확도 낮음) → 기각.

## R5. RAG 안내(챗봇) 구조

- **Decision**: 안전 규정집·민원 행정 양식·과거 처리 이력을 **임베딩하여 pgvector**에 저장, 질의 시 검색 결과를 컨텍스트로 LLM이 답변(FR-018/019). 근거 부재 시 추측 금지·담당 창구 안내. `paper-context-resolver` 스킬을 검색 패턴 참조로 사용.
- **Rationale**: 별도 벡터 DB 없이 PostgreSQL 단일 스택 유지(R1과 정합, 운영 단순). 출처 기반 답변으로 환각 억제.
- **Alternatives**: 외부 전용 벡터 DB(인프라 추가), LLM 파인튜닝(데이터·비용·갱신성 불리) → 기각.

## R6. 공문서 자동 생성

- **Decision**: 분류된 신고 + 테넌트별 양식 템플릿을 입력으로 **LLM이 공문서 초안 생성** → 담당자 검토·수정·확정(FR-015). 산출은 `docx`/`pdf` 스킬로 .docx/.pdf 출력.
- **Rationale**: 텍스트 타이핑 없는 행정 처리, 처리 시간 40% 단축(SC-004) 기여. 사람 확정 단계로 오류 통제.
- **Alternatives**: 규칙기반 템플릿 치환(유연성 부족), 완전 자동 발송(검증 부재 위험) → 기각.

## R7. 인증/세션 (하이브리드 신원)

- **Decision**: 시민 익명 신고는 **기기/세션 식별자**(서버 발급 토큰)로 추적, 선택적 회원은 **Supabase Auth**로 로그인하여 보상·이력·알림 수신(FR-024). 담당자/관리자는 Supabase Auth + 역할/테넌트 클레임 + RLS.
- **Rationale**: "3초 익명 신고" 진입장벽 0 + 로그인 부가가치 양립(Q1 결정).
- **Alternatives**: 전원 강제 로그인(진입장벽), 전면 익명(보상·알림 불가) → 기각.

## R8. 실시간 갱신·알림

- **Decision**: 관제 신규 신고/급증은 **Supabase Realtime**으로 대시보드에 푸시. 로그인 시민 상태 알림은 웹푸시/이메일(외부 메일 발송 MCP/서비스 바인딩)로 1분 이내 전달(SC-006/FR-017). 익명은 접수 번호 조회.
- **Rationale**: 폴링 대비 지연·부하 감소, 재난 급증 대응(엣지 케이스) 용이.
- **Alternatives**: 클라이언트 폴링(부하·지연), 푸시 전용(이메일 백업 부재) → 기각.

## R9. 데이터 보관·파기(1년 TTL)

- **Decision**: 처리 완료 신고는 **완료 시각 + 1년** 후 자동 파기 배치(사진·식별 데이터 삭제), 위험유형·위치·결과는 **비식별 집계 테이블**로만 영구 보존(FR-010a/SC 보관 결정).
- **Rationale**: 개인정보 최소보관 + 통계/리포트(후속 B2B) 양립.
- **Alternatives**: 무기한 보관(법적·윤리 리스크), 즉시 파기(추적·통계 손실) → 기각.

## R10. 확장성/가용성(99.9%)

- **Decision**: 웹/API는 **수평 확장 가능한 무상태 컨테이너**, AI 추론은 **GPU 노드 큐 기반 비동기 처리**(분류 결과는 3초 내 응답 목표, 무거운 후처리는 큐). DB는 관리형(Supabase) 이중화. 월 99.9% 목표(SC-011).
- **Rationale**: 동시 1만(SC-010)·재난 급증 흡수, 단일 장애점 최소화.
- **Alternatives**: 단일 인스턴스(가용성·확장성 미달), 전 구간 동기 추론(지연 폭증) → 기각.
