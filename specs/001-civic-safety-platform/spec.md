## Overview

시민이 현장에서 촬영한 한 장의 사진만으로 도시의 위험 요소(도로 파손, 시설물 훼손, 안전 위협 등)를 신고할 수 있고, 시스템이 사진을 자동으로 분석·분류하여 표준 신고서를 생성하고 담당 행정 부서로 전달하는 웹 기반 서비스다. 시민은 텍스트 입력 없이 빠르게 신고하고 처리 상황을 안내받으며, 지자체 담당자는 자동 분류·우선순위가 매겨진 민원을 관제 대시보드에서 확인·처리한다. 본 명세는 이 플랫폼을 논리적·체계적으로 구축하기 위한 사용자 가치, 기능 요구사항, 측정 가능한 성공 기준을 정의한다.

## Clarifications

### Session 2026-06-19

- Q: 시민이 신고하려면 계정이 필요한가? → A: 하이브리드. 시민은 계정 없이 사진+위치만으로 익명 신고할 수 있고, 접수 번호로 처리 상태를 조회한다. 회원가입/로그인한 시민에게는 추가로 참여 보상 적립, 개인 신고 이력 조회, 처리 상태 알림을 제공한다. 중복·남용 방지는 익명 신고의 경우 기기/세션 단위로, 로그인 신고의 경우 계정 단위로 수행한다.
- Q: 시스템은 멀티테넌트(여러 지자체)인가, 단일 지자체 전용인가? → A: 멀티테넌트로 설계한다. 지자체(테넌트)별로 데이터 격리, 권한 범위, 위험 유형·담당 부서 매핑, 운영 설정을 분리하며, 1차 파일럿은 단일 지자체로 운영한다.
- Q: 목표 서비스 가용성(uptime)은? → A: 월 99.9%(다운타임 월 약 43분). 공공 안전 서비스 표준 수준으로 설정한다.
- Q: 신고 데이터·사진의 보관 기간은? → A: 처리 완료 후 1년 보관 뒤 자동 파기한다. 단, 위험 유형·위치·처리 통계는 개인 식별 정보를 제거한 비식별 집계 형태로만 영구 보존한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 사진 한 장으로 위험 신고 (Priority: P1)

시민이 길에서 포트홀, 파손된 보도블록, 쓰러진 표지판 등 위험 요소를 발견하면, 모바일 웹에서 사진을 촬영(또는 업로드)하고 위치를 확인한 뒤 신고를 제출한다. 시스템은 사진을 자동 분석하여 위험 유형을 분류하고 위치 정보를 결합해 표준 신고서를 생성한 후 접수 번호를 부여한다. 시민은 텍스트를 한 글자도 입력하지 않고도 신고를 완료할 수 있다.

**Why this priority**: 이 흐름이 플랫폼의 핵심 가치("텍스트 타이핑 없는 3초 민원")이며, 단독으로도 시민이 위험을 신고하고 행정에 전달되는 완결된 가치를 제공하는 최소 기능 제품(MVP)이다.

**Independent Test**: 사진 1장과 위치만으로 신고를 제출하여 접수 번호와 자동 분류된 위험 유형이 반환되는지 확인함으로써 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 시민이 위험 요소 앞에 있는 상태에서, **When** 사진을 촬영하고 위치를 확인한 뒤 신고를 제출하면, **Then** 시스템은 위험 유형을 자동 분류하고 접수 번호가 포함된 신고 완료 확인을 3초 이내에 보여준다.
2. **Given** 위치 정보(GPS)가 사진에 없는 상태에서, **When** 시민이 신고를 시도하면, **Then** 시스템은 지도에서 위치를 직접 지정하도록 안내하고 위치 확정 후에만 제출을 허용한다.
3. **Given** 시스템이 사진의 위험 유형을 확신하지 못하는 상태에서, **When** 신고가 제출되면, **Then** 시스템은 추정 유형 후보를 제시하고 시민이 선택/수정할 수 있게 한 뒤 접수한다.
4. **Given** 신고가 접수된 상태에서, **When** 시민이 사진에 사람 얼굴이나 차량 번호판이 포함된 사진을 올리면, **Then** 시스템은 개인 식별 정보를 가린(비식별 처리) 상태로 신고를 저장한다.

### User Story 2 - 지자체 담당자의 관제 및 처리 (Priority: P1)

지자체 담당 공무원이 관제 대시보드에 로그인하면, 자동 분류·우선순위가 매겨진 신고 목록을 지도와 리스트로 확인한다. 각 신고의 상세(사진, 위치, 자동 분류 결과, 유사·중복 신고 묶음)를 검토하고, 담당 부서 배정·처리 상태 변경·자동 생성된 공문서 초안 확인을 수행한다. 처리 상태 변경은 신고한 시민에게 자동으로 안내된다.

**Why this priority**: 신고가 실제 행정 처리로 이어져야 플랫폼이 B2G 가치("처리 시간 단축")를 완성한다. 시민 신고(P1)와 함께 양면 시장의 다른 한 축을 이루는 필수 기능이다.

**Independent Test**: 담당자 계정으로 접수된 신고를 조회하고 상태를 "접수→처리중→완료"로 변경했을 때 목록·지도·시민 알림에 반영되는지 확인하여 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 신규 신고가 접수된 상태에서, **When** 담당자가 대시보드를 열면, **Then** 위험 유형·위치·접수 시각·우선순위가 표시된 신고가 목록과 지도에 나타난다.
2. **Given** 동일 위치/유형의 신고가 여러 건 접수된 상태에서, **When** 담당자가 목록을 보면, **Then** 시스템은 중복·유사 신고를 하나의 묶음으로 묶어 표시하고 건수를 함께 보여준다.
3. **Given** 담당자가 신고를 검토하는 상태에서, **When** 처리 상태를 변경하면, **Then** 변경 이력이 기록되고 해당 신고를 제출한 시민에게 상태 변경이 자동 안내된다.
4. **Given** 신고가 특정 위험 유형으로 분류된 상태에서, **When** 담당자가 공문서 생성을 요청하면, **Then** 시스템은 해당 민원에 맞는 표준 행정 문서 초안을 자동 생성하여 담당자가 검토·수정할 수 있게 한다.

### User Story 3 - AI 대화형 신고 안내 및 진행 상황 확인 (Priority: P2)

시민이 신고 과정이나 진행 상황에 대해 궁금할 때, 대화형 안내(챗봇)를 통해 "내 신고 어떻게 됐나요?", "이건 어디 부서가 처리하나요?" 같은 질문을 자연어로 묻고 답을 받는다. 안내는 과거 처리 이력과 행정 규정·안전 지표를 근거로 친절하게 설명한다.

**Why this priority**: 시민 경험을 크게 향상시키고 재참여를 유도하지만, 신고 접수·처리(P1)가 동작한 뒤에 가치가 있으므로 P2다.

**Independent Test**: 접수된 신고 번호로 진행 상황을 질문했을 때 정확한 현재 상태와 다음 단계를 자연어로 안내받는지 확인하여 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 시민이 제출한 신고가 있는 상태에서, **When** 대화 안내에 진행 상황을 질문하면, **Then** 시스템은 해당 신고의 현재 상태와 예상 다음 단계를 자연어로 안내한다.
2. **Given** 시민이 어떤 위험을 신고해야 할지 모르는 상태에서, **When** 상황을 설명하며 질문하면, **Then** 시스템은 적합한 위험 유형과 신고 방법을 안내한다.
3. **Given** 시스템이 질문에 대한 근거 자료가 없는 상태에서, **When** 답을 생성하면, **Then** 시스템은 추측 대신 "확인이 어렵다"는 안내와 담당 창구를 제시한다.

---

### User Story 4 - 신고 신뢰도 관리 및 시민 참여 보상 (Priority: P3)

플랫폼은 허위·중복·장난성 신고를 걸러내 행정력 낭비를 방지하고, 실제로 도움이 된 신고에 대해 **로그인한 시민**에게 참여 보상(포인트/마일리지 등)을 제공하여 건전한 참여 생태계를 유지한다. 익명 신고는 보상 대상이 아니며, 신뢰도 관리는 익명 신고의 경우 기기/세션·신고 내용·위치 기반으로, 로그인 신고의 경우 계정 단위로 수행한다.

**Why this priority**: 장기적 참여 활성화와 데이터 품질에 중요하지만, 핵심 신고/처리 흐름 없이는 의미가 없어 P3다.

**Independent Test**: 동일 위치·유형의 사진을 반복 제출했을 때 중복으로 묶이거나 차단되고, 로그인 시민의 처리 완료된 유효 신고에 보상이 적립되는지 확인하여 독립적으로 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 이미 신고된 동일 위치·유형을 다시 신고하는 상태에서, **When** 제출하면, **Then** 시스템은 기존 신고 묶음에 합류시키고 중복으로 표시한다(중복 보상 미적립).
2. **Given** 로그인한 시민의 신고가 담당자에 의해 "유효·처리 완료"로 확정된 상태에서, **When** 처리가 완료되면, **Then** 시스템은 해당 시민에게 정해진 참여 보상을 적립한다.
3. **Given** 한 기기/세션 또는 계정이 짧은 시간에 비정상적으로 많은 신고를 제출하는 상태에서, **When** 임계치를 초과하면, **Then** 시스템은 추가 제출을 제한하고 검토 대상으로 표시한다.
4. **Given** 담당자가 신고를 검토하는 상태에서, **When** 허위/장난성으로 판정하면, **Then** 시스템은 해당 신고를 무효 처리하고 보상 적립을 막으며 처리 통계에서 제외한다.

---

### Edge Cases

- 야간·폭우·저화질 등으로 사진 분석 신뢰도가 낮을 때: 시스템은 자동 분류를 강요하지 않고 시민이 유형을 직접 선택/수정하도록 유도한다.
- 사진에 위험 요소가 없거나 분류 불가능할 때: 시스템은 신고를 반려하거나 "기타"로 접수하고 담당자 수동 검토 큐로 보낸다.
- 위치가 해당 지자체 관할 밖일 때: 시스템은 관할 외임을 안내하고 적절한 기관/창구로 안내한다.
- 네트워크가 불안정하여 업로드가 중단될 때: 시스템은 작성 중 신고를 임시 저장하고 연결 복구 시 이어서 제출할 수 있게 한다.
- 동일 위험에 대한 대량 신고가 단시간에 몰릴 때(예: 재난 상황): 시스템은 묶음 처리하고 담당자에게 급증 알림을 제공한다.
- 사진에 타인의 얼굴/번호판/주소 등 개인정보가 포함될 때: 시스템은 비식별 처리 후 저장하고 원본 노출을 방지한다.
- 처리 완료 후 시민이 동일 위험의 재발을 신고할 때: 시스템은 신규 신고로 접수하되 과거 이력과 연결해 표시한다.

## Requirements *(mandatory)*

### Functional Requirements

#### 신고 접수 (시민)

- **FR-001**: 시스템은 시민이 사진을 촬영하거나 업로드하여 위험 신고를 시작할 수 있도록 해야 한다.
- **FR-002**: 시스템은 신고 사진의 위치 정보(좌표)와 촬영 시각을 수집해야 하며, 위치 정보가 없을 경우 지도에서 위치를 직접 지정하도록 해야 한다.
- **FR-003**: 시스템은 업로드된 사진을 자동 분석하여 위험 유형을 분류하고, 분류 결과를 시민에게 제시해야 한다.
- **FR-004**: 시스템은 자동 분류 신뢰도가 낮을 때 후보 유형을 제시하고 시민이 선택·수정한 뒤 제출할 수 있도록 해야 한다.
- **FR-005**: 시스템은 시민이 텍스트를 입력하지 않고도(사진+위치만으로) 신고를 완료할 수 있도록 해야 한다.
- **FR-006**: 시스템은 신고 접수 시 고유 접수 번호를 부여하고 접수 완료 확인을 시민에게 제공해야 한다.
- **FR-007**: 시스템은 접수된 신고로부터 담당 부서로 전달 가능한 표준 신고서를 자동 생성해야 한다.

#### 개인정보 및 안전

- **FR-008**: 시스템은 신고 사진에 포함된 사람 얼굴·차량 번호판 등 개인 식별 정보를 비식별 처리(가림)한 상태로 저장해야 한다.
- **FR-009**: 시스템은 시민의 개인정보·위치정보를 신고 처리 목적 범위 내에서만 사용하고, 관련 법규(개인정보보호 의무)를 준수해야 한다.
- **FR-010**: 시스템은 익명 시민이 접수 번호로 자신이 제출한 신고의 처리 상태를 조회할 수 있도록 하고, 로그인한 시민에게는 본인이 제출한 신고의 개인 이력 목록을 제공해야 한다.
- **FR-010a**: 시스템은 처리 완료된 신고 데이터(사진 포함)를 완료 후 1년간 보관한 뒤 자동 파기해야 하며, 파기 이후에는 개인 식별 정보를 제거한 비식별 집계 통계(위험 유형·위치·처리 결과 등)만 보존해야 한다.

#### 관제 및 처리 (지자체 담당자)

- **FR-011**: 시스템은 담당자가 접수된 신고를 목록과 지도 형태로 조회할 수 있는 관제 대시보드를 제공해야 한다.
- **FR-012**: 시스템은 신고를 위험 유형·위치·시급성 기준으로 우선순위화하여 표시해야 한다.
- **FR-013**: 시스템은 동일 위치·유형의 중복/유사 신고를 자동으로 묶어 하나의 처리 단위로 표시해야 한다.
- **FR-014**: 시스템은 담당자가 신고를 담당 부서에 배정하고 처리 상태(예: 접수/처리중/완료/반려)를 변경할 수 있도록 해야 한다.
- **FR-015**: 시스템은 신고 유형에 맞는 표준 행정 공문서 초안을 자동 생성하고 담당자가 검토·수정·확정할 수 있도록 해야 한다.
- **FR-016**: 시스템은 모든 상태 변경과 처리 이력을 시각·담당자와 함께 기록(감사 추적)해야 한다.

#### 알림 및 대화형 안내

- **FR-017**: 시스템은 신고 처리 상태가 변경될 때, 로그인한 시민에게는 자동 알림(푸시/이메일 등)을 전달하고, 익명 시민에게는 접수 번호 기반 상태 조회로 변경 사항을 확인할 수 있도록 해야 한다.
- **FR-018**: 시스템은 시민이 신고 방법·진행 상황·담당 부서 등을 자연어로 질의하고 안내받을 수 있는 대화형 안내 기능을 제공해야 한다.
- **FR-019**: 시스템은 대화형 안내가 근거 자료(처리 이력·행정 규정·안전 지표)에 기반해 답하도록 하고, 근거가 없을 때는 추측하지 않고 담당 창구를 안내해야 한다.

#### 신뢰도 및 참여 보상 관리

- **FR-020**: 시스템은 동일 위치·유형의 반복 신고를 신고 내용·위치 기반으로 중복 식별하여 기존 신고 묶음에 합류시켜야 한다.
- **FR-021**: 시스템은 단시간 내 비정상적으로 많은 신고를 제출하는 기기/세션(익명) 또는 계정(로그인)을 제한하고 검토 대상으로 표시해야 한다.
- **FR-022**: 시스템은 담당자가 신고를 허위/장난성으로 판정하여 무효 처리하고 보상 적립을 막으며 처리 통계에서 제외할 수 있도록 해야 한다.
- **FR-023**: 시스템은 담당자가 "유효·처리 완료"로 확정한 신고가 로그인한 시민이 제출한 것일 경우, 해당 시민에게 정해진 참여 보상을 적립해야 한다(익명 신고는 보상 대상에서 제외).

#### 계정 및 권한

- **FR-024**: 시스템은 시민이 계정 없이 익명으로 신고할 수 있도록 하고, 선택적으로 회원가입/로그인한 시민에게는 보상 적립·개인 이력 조회·처리 알림을 제공해야 하며, 지자체 담당자와 관리자에 대해서는 인증 및 역할별 접근 제한을 적용해야 한다.
- **FR-025**: 시스템은 담당자가 자신의 관할 지역/부서 범위의 신고만 조회·처리하도록 권한을 제한해야 한다.

#### 멀티테넌시 (지자체)

- **FR-026**: 시스템은 여러 지자체(테넌트)를 동시에 수용하고, 테넌트별로 신고·담당자·처리 이력·공문서 등 데이터를 논리적으로 격리해야 한다(한 지자체 담당자가 다른 지자체의 데이터에 접근 불가).
- **FR-027**: 시스템은 테넌트별로 위험 유형 분류 체계, 담당 부서 매핑, 공문서 양식, 참여 보상 정책 등 운영 설정을 독립적으로 구성할 수 있도록 해야 한다.
- **FR-028**: 시스템은 시민 신고의 위치 좌표를 기준으로 관할 지자체(테넌트)를 판별하여 해당 테넌트로 라우팅해야 하며, 어느 테넌트 관할에도 속하지 않는 신고는 관할 외로 안내해야 한다.

### Key Entities *(include if feature involves data)*

- **신고(Report)**: 시민이 제출한 단일 위험 신고. 접수 번호, 사진(비식별 처리본), 위치 좌표, 촬영/접수 시각, 자동 분류된 위험 유형, 신뢰도, 처리 상태, 우선순위, 묶음 식별자, 관할 지자체(테넌트) 참조, 그리고 제출 출처 식별자(익명 신고는 기기/세션 식별자, 로그인 신고는 시민 계정 참조)를 가진다.
- **지자체(Tenant/Municipality)**: 서비스를 구독·운영하는 행정 주체(구청/시청 등). 관할 경계, 위험 유형·부서 매핑·공문서 양식·보상 정책 등 테넌트별 설정, 소속 담당자 집합을 가진다. 모든 신고·담당자·처리 이력·공문서는 하나의 지자체에 귀속된다.
- **위험 유형(Hazard Category)**: 지자체별로 구성되는 신고 분류 체계(예: 도로 파손, 시설물 훼손, 안전 위협 등). 소속 지자체, 담당 부서 매핑 정보를 가진다.
- **시민 계정(Citizen Account)**: 선택적으로 회원가입한 시민. 인증 정보, 본인 신고 이력 참조, 참여 보상 잔액, 신뢰도/제한 상태를 가진다(익명 신고는 계정 없이 기기/세션 식별자로만 추적).
- **담당자(Municipal Officer)**: 신고를 검토·처리하는 행정 사용자. 소속 지자체(테넌트), 소속 부서, 관할 범위, 권한 역할을 가진다.
- **처리 이력(Processing Log)**: 신고의 상태 변경·배정·문서 생성 등 모든 행위 기록. 행위자, 시각, 변경 내용을 가진다.
- **공문서(Official Document)**: 신고로부터 자동 생성된 표준 행정 문서 초안. 연결된 신고, 생성 시각, 검토/확정 상태를 가진다.
- **알림(Notification)**: 로그인한 시민에게 전달되는 상태 변경·보상 안내. 수신 시민 계정, 신고 참조, 내용, 전달 시각을 가진다.
- **참여 보상(Reward)**: 로그인 시민의 유효 신고에 대해 적립되는 포인트/마일리지. 시민 계정, 연결된 신고, 적립 시각, 금액/포인트를 가진다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 시민이 사진 촬영부터 신고 제출 완료까지 평균 30초 이내, 단순 신고는 최소 입력으로 "3초 신고" 경험을 달성한다(텍스트 입력 0건으로 제출 가능).
- **SC-002**: 신고 제출 후 자동 분류 결과와 접수 확인이 3초 이내에 시민에게 표시된다.
- **SC-003**: 자동 위험 분류의 정확도가 적정 촬영 조건에서 85% 이상이다(담당자 확정 기준 대비).
- **SC-004**: 지자체의 민원 분류 및 초기 처리 소요 시간이 도입 전 대비 40% 단축된다.
- **SC-005**: 접수된 신고 중 중복·유사 신고 묶음 처리율이 90% 이상으로, 담당자가 중복 건을 개별 처리하지 않는다.
- **SC-006**: 처리 상태 변경 시 시민 알림이 1분 이내에 전달된다.
- **SC-007**: 신규 시민이 별도 안내 없이 첫 신고를 성공적으로 완료하는 비율이 90% 이상이다.
- **SC-008**: 허위·장난성으로 최종 판정되는 신고 비율이 전체의 5% 이하로 유지된다.
- **SC-009**: 개인정보가 포함된 사진의 비식별 처리 적용률이 100%이다(원본 식별 정보가 저장·노출되지 않음).
- **SC-010**: 동시 접속 시민 10,000명 환경에서 신고 제출 성공률 99% 이상을 유지한다.
- **SC-011**: 서비스 가용성을 월 99.9% 이상(월 다운타임 약 43분 이내) 유지한다.

## Assumptions

- 본 서비스의 1차 범위는 **웹 기반(모바일 웹 포함)** 서비스이며, 네이티브 앱은 후속 단계로 둔다.
- 시민은 위치 정보 제공이 가능한 환경(GPS 또는 수동 위치 지정)에서 신고한다고 가정한다.
- 지자체와의 연동은 우선 플랫폼 내 표준 신고서·공문서 생성 및 대시보드 처리로 완결하며, 외부 행정 시스템(안전신문고 등)과의 실시간 API 연동은 연동 규격 확보 후 단계적으로 적용한다.
- 개인정보 비식별 처리 기준은 얼굴·차량 번호판을 기본 대상으로 하며, 관련 법규의 산업 표준을 따른다.
- 시스템은 멀티테넌트로 설계하여 지자체별 데이터 격리·설정을 지원하되, 1차 파일럿은 단일 지자체로 운영한다.
- 위험 유형 분류 체계와 담당 부서 매핑은 지자체(테넌트)별로 도입 지자체의 민원 처리 기준에 따라 초기 구성한다.
- 신고의 관할 지자체 판별은 위치 좌표와 지자체 관할 경계 데이터를 기준으로 한다.
- 참여 보상의 구체적 단위·재원(지역화폐/마일리지 등)은 도입 지자체 정책에 따라 설정 가능한 것으로 가정한다.
- 인증은 웹 서비스 표준 방식(세션 기반 또는 OAuth2 등)을 사용하며, 시민/담당자/관리자 역할을 구분한다.
- 신고 데이터·사진은 처리 완료 후 1년 보관 뒤 자동 파기하며, 이후에는 비식별 집계 통계만 영구 보존한다(개인정보 관련 법규 준수).
- B2B 데이터 리포트 판매, 정부 과제·바우처 등 수익 모델은 본 기능 명세의 직접 범위 밖이며, 비식별 데이터 집계는 후속 기능으로 둔다.
